# Phase 5.2 Context: SBC Refinements and Fixes

## Overview
This phase addresses follow-up issues from the initial implementation of unaligned SAM/BAM/CRAM (SBC) support. It includes fixing the BAM-to-FASTA conversion, refining CLI options for better clarity, and ensuring documentation consistently emphasizes that SBC support is for *unaligned* data.

## Implementation Decisions

### 1. Output Format Flag
- **Decision:** Add an explicit `--output-format` (or `-O`) option to the `reads` subcommand.
- **Allowed Values:** `fasta`, `fastq`, `sam`, `bam`, `cram`.
- **Rationale:** While format inference from file extensions works well, an explicit flag is needed to override the default behavior, especially when outputting to `stdout` where no extension is available.
- **Impact:** This flag should be used to explicitly drive the output writer format. If `fasta` is specified (either via extension or this flag), it must force the `AlignmentSource` to output FASTA, fixing the current bug where it defaults to FASTQ if quality scores are present.

### 2. CLI Option Naming for Compression
- **Decision:** Rename the existing `--output-type` option (which handles compression formats like `u, b, g, l, x, z`) to `--compress-type`.
- **Rationale:** The term "output-type" is too ambiguous now that the tool supports multiple file formats (SBC vs sequence). Renaming it to `--compress-type` clarifies that it specifically controls the *compression algorithm*, while the new `--output-format` controls the *file format*.
- **Impact:** This is a breaking change for the CLI, but necessary for long-term clarity. The short flag `-O` should be reassigned to `--output-format`, and a new short flag (e.g., `-Z` or none) should be used for `--compress-type`.

### 3. FASTA/FASTQ Inference Fix
- **Decision:** When writing an alignment record to sequence format, the decision to write FASTA or FASTQ should strictly follow the `--output-format` flag or the output file extension. 
- **Rationale:** Currently, `AlignmentSource` outputs FASTQ if quality scores exist, even if the user wanted FASTA. The explicit format selection must override this heuristic.

### 4. Documentation Clarity
- **Decision:** All references to SAM/BAM/CRAM support in the `reads` subcommand documentation (README, CLI help strings, project docs) must be qualified with "unaligned".
- **Rationale:** To prevent user confusion and explicitly state that `rasusa reads` is not intended for random subsampling of mapped alignments (which is handled by `rasusa aln`).

## Scope Boundaries
- **No FASTQ to BAM Conversion:** We maintain the restriction that sequence formats cannot be converted into alignment formats. This remains out of scope.
- **No Automatic Sorting/Indexing:** Any uBAM output will remain unindexed and in whatever order it was subsampled.

## Next Steps for Downstream Agents
1. **Planner:** The planner should update the `PLAN.md` for this phase to incorporate these specific CLI changes (renaming `--output-type`, adding `--output-format`) and the logic updates for FASTA generation.
2. **Executor:** The executor will modify `src/reads.rs`, `src/cli.rs`, and `src/source.rs` according to these decisions, and update tests/docs accordingly.
