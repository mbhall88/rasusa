# Plan 05.1-01 Summary: SBC Output Refinement and Validation

## Tasks Completed
1. **Refine SBC Output and Headers**: 
    - Updated `src/reads.rs` to default the output format to the input alignment format (SAM/BAM/CRAM) for SBC inputs, especially when outputting to stdout.
    - Implemented flexible format matching: SBC to FASTQ conversion is supported via file extension, while FASTQ to SBC conversion is explicitly blocked with a clear error message.
    - Added a `program_entry` helper in `src/source.rs` and integrated it into `AlignmentSource::filter_reads_into` to ensure all SBC outputs contain a 'rasusa' `@PG` header record with version and command-line information.
2. **Implement Unmapped Validation**:
    - Added a check in `AlignmentSource::read_lengths` and `AlignmentSource::filter_reads_into` to verify the `UNMAPPED` flag (0x04) for all alignment records.
    - Encountering a mapped read now triggers a fatal error: "Error: Mapped read detected, please use `rasusa aln` for aligned data".
    - Updated existing unit tests in `src/source.rs` to use unmapped flags to comply with the new validation logic.
3. **Integration Testing**:
    - Added 6 new integration tests in `tests/main.rs` using dedicated uBAM test cases from `tests/cases/ubam/`.
    - Verified single-end and paired-end uBAM processing, SBC to FASTQ conversion, and error handling for mapped BAMs and unsupported FASTQ-to-BAM conversions.
    - Fixed the `reads_reproducibility` integration test to use record-level comparison, ensuring it passes despite path differences in the header command line.

## Verification
- `cargo test` confirms all 147 unit tests and 28 integration tests pass.
- Manual inspection of BAM headers confirms correct 'rasusa' `@PG` entries.
- Validated that mapped BAM files are correctly rejected by the `reads` subcommand.

## Conclusion
The unaligned SAM/BAM/CRAM support in the `reads` subcommand is now refined, robust, and follows standard bioinformatics conventions for header traceability and data integrity.
